## Задание

**Ожидаемый результат:** тесты покрывают ключевую бизнес-логику и поведение UI-компонентов.

> ⚠️ Необязательно покрывать тестами абсолютно всё. Ваша задача — отработать умение написания тестов и улучшить качество фронтенда. Можете использовать это домашнее задание, чтобы доработать предыдущие домашние задания и итоговый проект.

### 1. Настройка окружения для тестов

Подготовьте проект к запуску тестов:

- установите и настройте **Vitest** (рекомендуется для Vite-проектов) или **Jest**;
- подключите React Testing Library: `@testing-library/react`, `@testing-library/user-event`, `@testing-library/jest-dom`;
- добавьте скрипт `"test"` в `package.json`;
- создайте конфигурационный файл (`vitest.config.ts` или `jest.config.ts`) с поддержкой `jsdom`-окружения.

### 2. Юнит-тесты на reducer

Напишите тесты для логики стейт-менеджера:

- `ADD_MESSAGE` — после добавления сообщения массив `messages` увеличивается на 1, новое сообщение находится в конце.
- `CREATE_CHAT` — создаётся новый чат с уникальным `id`, он появляется в массиве `chats`.
- `DELETE_CHAT` — чат удаляется из массива, при удалении активного чата `activeChatId` сбрасывается.
- *(Дополнительно)* `RENAME_CHAT` — название чата корректно обновляется по `id`.

### 3. Компонентные тесты через React Testing Library

#### `InputArea`
- при непустом значении и клике на кнопку «Отправить» вызывается `onSend` с текстом сообщения;
- при нажатии `Enter` с непустым вводом вызывается `onSend`;
- кнопка «Отправить» заблокирована (`disabled`) при пустом поле ввода.

#### `Message`
- при `variant="user"` — компонент содержит текст сообщения и CSS-класс, соответствующий пользователю;
- при `variant="assistant"` — компонент содержит текст сообщения и CSS-класс, соответствующий ассистенту;
- кнопка «Копировать» присутствует только для `variant="assistant"`.

#### `Sidebar`
- при вводе текста в поле поиска список чатов фильтруется по названию;
- при пустом поисковом запросе отображаются все чаты;
- при нажатии кнопки «Удалить» появляется запрос на подтверждение.

### 4. Мок `localStorage`

Протестируйте персистентность данных без реального браузерного окружения:

- мокируйте `localStorage` через `vi.stubGlobal` (Vitest) или `jest.spyOn`;
- проверьте, что:
  - при изменении стейта данные сохраняются в `localStorage`;
  - при инициализации приложения данные корректно восстанавливаются из `localStorage`;
  - при «битых» данных в `localStorage` приложение не падает (невалидный JSON).

## Технические требования

- Тестовый фреймворк: **Vitest** (для Vite) или **Jest** — на выбор.
- **React Testing Library** для компонентных тестов.
- Тесты не должны делать реальных запросов к GigaChat API — мокировать через `vi.mock` / `jest.mock`.
- Каждый тест изолирован: не зависит от порядка запуска и результатов других тестов.
- Тесты находятся рядом с тестируемым файлом (`.test.tsx` / `.test.ts`).
